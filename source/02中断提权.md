# 1

Q: IDT 是什么？

A: IDT（Interrupt Descriptor Table，中断描述符表）是 x86 保护模式下 CPU 用于查找中断/异常处理程序的表。每个中断向量号对应表中一个门描述符，CPU 根据描述符中的信息跳转到对应的处理函数，并可能切换特权级。

GROUP: 周壑内核 02

# 2

Q: 除零异常（#00）的触发原因和异常类别是什么？

A: 触发原因：执行 DIV / IDIV 指令时除数为 0。异常类别为 Fault（错误），通常无法安全继续执行，需修正逻辑或终止进程。Windows 异常码为 0xC0000094（STATUS_INTEGER_DIVIDE_BY_ZERO）。

GROUP: 周壑内核 02

# 3

Q: 调试异常（#01）的触发原因有哪些？

A: 四种触发源：
1. 单步调试（TF 标志置位）
2. 硬件断点命中
3. 执行 INT 1 指令
4. 数据断点或 IO 断点

异常类别为 Fault 或 Trap，取决于具体触发源，多数情况下可继续执行。

GROUP: 周壑内核 02

# 4

Q: x86 异常中 Fault 和 Trap 的区别是什么？

A:
- Fault：异常发生在指令执行前或执行中，处理完后 CPU 重新执行该指令（如页错误）
- Trap：异常发生在指令执行后，处理完后 CPU 继续执行下一条指令（如单步调试）

除零异常是 Fault，调试异常可能是 Fault 或 Trap。

GROUP: 周壑内核 02

# 5

Q: 什么是中断门（Interrupt Gate）？

A: 中断门是 IDT 中的一种门描述符类型，是 CPU 提供的"合法"提权通道。当通过 int n 指令触发中断时，CPU 检查当前 CPL 与门描述符 DPL 的关系，满足条件则自动将 CPL 提升到描述符中 Segment Selector 所指的特权级（通常是 Ring 0）。

GROUP: 周壑内核 02

# 6

Q: 中断门描述符由哪些关键字段组成？

A: 8 字节结构的关键字段：
- Offset（分高低 16 位）：处理函数地址
- Segment Selector：目标代码段选择子（决定提权后的特权级）
- P（Present）：描述符是否有效
- DPL：门的特权级，决定谁能调用此门
- Type：门类型（0xE = 32 位中断门）
- D（Size）：0=16 位，1=32 位

GROUP: 周壑内核 02

# 7

Q: 中断门中 Segment Selector 字段的作用是什么？

A: 指定处理函数所在代码段的选择子。通常设为 0x0008，即内核代码段（Ring 0），这样 CPU 提权后就在 Ring 0 执行处理函数。这个选择子决定了提权后的 CPL。

GROUP: 周壑内核 02

# 8

Q: 中断门中 DPL 字段的作用是什么？

A: DPL（Descriptor Privilege Level）决定哪些特权级的代码有资格通过此门。CPU 检查规则为：CPL ≤ DPL（数字上）。DPL = 3 时，Ring 3 程序可以触发；DPL = 0 时，只有 Ring 0 可以触发，Ring 3 触发会引发 #GP（通用保护异常）。

GROUP: 周壑内核 02

# 9

Q: int 指令触发的中断和除零异常走的路径有何不同？

A: int n（如 int 0x0 在三环执行）和真正的除零异常（CPU 执行 DIV 时除数为 0 自动触发）走的是不同的路径：
- 除零异常：CPU 硬件在遇到除零时自动触发，#00，无视当前 CPL，直接走 IDT[0]
- int 0x0（三环）：软件中断指令，需要通过 DPL 检查，若 IDT[0] 的 DPL 不满足条件则触发 #GP

GROUP: 周壑内核 02

# 10

Q: 为什么构造 int 0x20 时需要将 DPL 设为 3？

A: 因为用户态程序在 Ring 3 执行 int 0x20 时，CPU 会做特权检查：CPL(3) ≤ DPL。如果 DPL = 0，CPL(3) > DPL(0)，检查失败，触发 #GP。DPL = 3 时，CPL(3) ≤ DPL(3)，检查通过，允许进入。DPL=3 是整个提权实验能成功的核心前提。

GROUP: 周壑内核 02

# 11

Q: 中断门提权的完整执行流程是怎样的？

A:
1. Ring 3 程序执行 int 0x20
2. CPU 查 IDT，索引 0x20 找到门描述符
3. 特权检查通过（CPL=3 ≤ DPL=3）
4. CPU 自动切换栈、保存 SS/ESP/EFLAGS/CS/EIP，将 CPL 提升到 0
5. 跳转到门描述符指定的处理函数（Ring 0 执行）
6. 处理函数执行内核操作（如读取内核内存）
7. iretd 从栈恢复上下文，返回 Ring 3

GROUP: 周壑内核 02

# 12

Q: 为什么进入 Ring 0 后可以直接读取 0x80b99400 等内核地址？

A: Ring 0（CPL=0）下，CPU 不再受 Ring 3 的页表和段限限制。只要该内核地址的 PTE 允许内核模式访问，Ring 0 代码就可以直接读取。0x80b99400 是 IDT 基地址附近的内核内存，Ring 0 下可正常访问。

GROUP: 周壑内核 02

# 13

Q: iretd 指令的作用是什么？

A: iretd（Interrupt Return Double-word）从当前栈中依次弹出 EIP、CS、EFLAGS、ESP、SS，恢复中断前的代码段、栈段和标志寄存器，使 CPU 优雅地从 Ring 0 降权回到 Ring 3，继续执行 int 指令之后的代码。

GROUP: 周壑内核 02

# 14

Q: 构造中断门时，描述符中的 Offset 地址是如何计算的？

A: Offset 分两部分存储：Offset[15:0] 在描述符低 16 位，Offset[31:16] 在描述符高 16 位。合并公式：完整地址 = Offset[31:16] << 16 | Offset[15:0]。如 0040ee00`00081000 中，Offset[31:16] = 0x0040，Offset[15:0] = 0x1000，合并为 0x00401000。

GROUP: 周壑内核 02

# 15

Q: 中断门描述符 DPL=3 和 DPL=0 的关键区别是什么？

A:
- DPL=0：只有 Ring 0 代码能通过此门，Ring 3 触发会引发 #GP。这是内核正常中断门的典型设置，保护内核入口。
- DPL=3：Ring 3 代码也能通过此门进入 Ring 0。这是本实验"欺骗" CPU 的关键——通过手动将 DPL 改为 3，让三环程序获得合法提权通道。

GROUP: 周壑内核 02

# 16

Q: 为什么本实验中 IdtEntry 函数必须声明为 __declspec(naked)？

A: naked 函数告诉编译器不生成标准的 prolog/epilog（如 push ebp / mov ebp, esp / pop ebp / ret），因为中断处理函数必须以 iretd 返回而非 ret。如果编译器插入 ret 指令，栈结构会被破坏，导致返回 Ring 3 时崩溃。

GROUP: 周壑内核 02

# 17

Q: 本实验"访问内存成功"的本质是什么？

A: 本质是通过手动构造 DPL=3 的中断门，绕过 CPU 特权检查，让三环程序通过 int 0x20 合法地进入 Ring 0。进入 Ring 0 后，CPU 不再施加用户态内存访问限制，可任意读取内核地址，最后通过 iretd 安全返回。这不是"漏洞利用"，而是利用了 CPU 原本就提供的合法提权机制（只不过正常的内核门 DPL 不会设成 3）。

GROUP: 周壑内核 02

# 18

Q: 什么情况下 IDT 表项会被"Hook"？

A: 当驱动程序或 Rootkit 修改了 IDT 中某个中断向量的入口地址，使其指向自己的代码而非原始处理函数时，称为 IDT Hook。笔记截图中 #12（Machine Check）被标记为 idt hook，说明该中断入口已被修改。
